### Remaining PF2e Findings

#### A1-4: Target Resolution Reliability (Carried Forward)
**Severity: 🟠 WRONG**

`_getTargets()` attempts:
1. `message.flags.pf2e.target.token` (single target UUID)
2. Fallback: `game.user.targets` (current selection)

**Analysis:**
- Path 1 only provides a **single** target. PF2e's `createChatMessage` flags store at most one target in `flags.pf2e.target`. Multi-target spells don't reliably store all targets here.
- Path 2 (`game.user.targets`) is unreliable after-the-fact.
- DnD5e (via Midi-QOL) gets reliable multi-target data from `workflow.hitTargets`/`workflow.saves`.

**Current impact:** For AoE spells, PF2e resolves 0-1 targets, falling to the "no targets" branch (generic `BLOODY_HIT`). Single-target attacks work when user has target selected.

**Status:** Unchanged from 2026-06-03. No PF2e API improvement identified. PF2e v6+ may expose `ChatMessage#targets` — monitor for future fix.

#### A1-5: VOCAL_STAGGER Default (1400ms)
**Severity: 🔵 INFO**

PF2e hardcoded fallback is 1400ms vs DnD5e's 400ms. When orchestrator is loaded, both use the same configurable default. The difference only matters if orchestrator fails to initialize. Intentional design for PF2e's later-resolving damage messages.

**Status:** Unchanged. Correct by design.

### Remaining Daggerheart Findings

#### A2-2: Content Keyword Matching (Carried Forward)
**Severity: 🟠 WRONG**

Result detection uses `msgContains(content, ["MISS", "FAILURE", "FAIL"])` which performs case-insensitive substring matching via `upper.indexOf(v.toUpperCase()) >= 0`.

**Problems:**
- **False positives**: "DISMISS" matches "MISS", "WATERFALL" matches "FAIL". No word-boundary detection in `msgContains()`.
- **Localization**: Non-English DH installs won't match keywords.
- **Format changes**: If DH changes from uppercase text results to icons/styled spans, detection breaks silently.

**Mitigating factor**: Primary detection uses DC comparison and flags. Content scanning is last-resort fallback when flags/DC are absent. The `msgContains` utility (`utils.js`) has no word-boundary logic — it's a simple `indexOf` check.

**Status:** Unchanged from 2026-06-03. The utility function remains a simple substring match.

#### A2-5: Item Resolution Chain Fragility (Carried Forward)
**Severity: 🟠 WRONG**

4-step resolution in `handleInfo()`:
1. UUID via roll options/flags (most reliable)
2. Exact name match on actor items
3. Fuzzy prefix match (item name starts with extracted prefix)
4. Raw title match

**Risk:** Step 3 can match wrong items (e.g. "Fire" prefix matches "Fireball" AND "Fire Shield"). As DH matures and adds UUIDs to all roll payloads, steps 2-4 become dead code that could still produce mismatches when UUID lookup fails.

**Status:** Unchanged from 2026-06-03. UUID path correctly prioritized; fallbacks remain as safety net.

#### A2-1: Two-Phase Render Timing
**Severity: 🔵 INFO (downgraded from 🟡)**

Phase 2 triggers when elapsed time exceeds 500ms. DH's current re-render happens at ~4 seconds. The gap is generous. Additionally, Phase 2 entries are now cleaned up after processing (no more unbounded growth). Risk assessment lowered.

**Status:** Improved (memory cleanup added). Timing threshold unchanged and appropriate.

### Remaining DH Stability Concern

#### D-2: Silent Path Failure (Carried Forward)
**Severity: 🟠 WRONG**

When DH releases a new version that changes data paths, the adapter's resource detection (stress, armor, hope) will silently stop working. The `validateSchema()` runs at startup but only warns about HP paths. The runtime `handlePreUpdate()` has a partial detection at line 468:

```js
} else if (JSON.stringify(changes).includes("stress")) {
    Logger.warn(`DH | Stress update detected in changes but no known path matched...`);
}
```

This pattern exists for stress and armor but is expensive (`JSON.stringify` on every actor update) and only fires when the string "stress"/"armor" appears somewhere in the changes object.

**Recommendation:** Add a one-time diagnostic on first successful path resolution that logs which path is live. This makes it obvious in logs which paths are active and provides breadcrumbs when debugging path staleness after a DH update.
