### Detailed PF2e Findings

#### A1-1: `createChatMessage` Filter Robustness
**Severity: 🟡 COSMETIC**

The `createChatMessage` hook fires for ALL chat messages. The PF2e adapter filters on `message.flags?.pf2e?.context?.type` checking for `"strike-attack-roll"`, `"spell-attack-roll"`, or `"damage-roll"`.

**Analysis:** This filter is robust for current PF2e versions. Skill checks use context type `"skill-check"`, saves use `"saving-throw"`, and flat checks use `"flat-check"`. These are all excluded by the explicit type matching. No false positives identified in the current code path.

**Risk:** If PF2e introduces new roll context types that contain "roll" in the name, they won't match since the check is exact-string equality, not substring. Safe.

#### A1-2: Degree of Success Numeric Mapping
**Severity: 🟡 COSMETIC**

`_normaliseOutcome()` maps numeric 0-3 → string outcomes:
```js
["criticalFailure", "failure", "success", "criticalSuccess"][raw]
```

**Analysis:** PF2e v5.x uses string outcomes in `context.outcome` (e.g. `"criticalSuccess"`). The numeric path (0-3) is a legacy compatibility path from PF2e v4.x. The adapter handles both, which is correct defensive coding. Current PF2e versions primarily deliver strings. The numeric mapping order (0=criticalFailure, 3=criticalSuccess) matches PF2e's `DegreeOfSuccess` enum.

**Risk:** Low. If PF2e v6 changes the enum values (unlikely for a stable system), the string path still works.

#### A1-3: Healing Detection Gaps
**Severity: 🟠 WRONG**

`_isHealing()` checks:
1. `item.system.traits.value` for `"healing"` trait
2. `message.flags.pf2e.damageRoll.instances` for all `type === "healing"` or `type === "temphp"`

**Analysis:** Missing coverage:
- **`vitality` damage type**: PF2e v5+ renamed "positive energy" healing to "vitality". The trait check catches items with the "healing" trait (which most healing spells have), but a raw vitality-typed damage roll without the healing trait would NOT be caught.
- The `_getSpellTraitKey()` correctly handles `vitality`/`positive` for spell sound selection, but `_isHealing()` does not check for `i.type === "vitality"` or `i.type === "positive"` in the instances array.

**Fix:** Add `i.type === "vitality" || i.type === "positive"` to the `damageInstances.every()` check.

#### A1-4: Target Resolution Reliability
**Severity: 🟠 WRONG**

`_getTargets()` attempts:
1. `message.flags.pf2e.target.token` (single target UUID)
2. Fallback: `game.user.targets` (current selection)

**Analysis:**
- Path 1 only provides a **single** target. PF2e's `createChatMessage` flags store at most one target in `flags.pf2e.target`. Multi-target spells don't reliably store all targets here.
- Path 2 (`game.user.targets`) is unreliable after-the-fact because targets may have changed between the roll and the hook firing (especially with automated workflows).
- DnD5e (via Midi-QOL) gets reliable multi-target data from `workflow.hitTargets` and `workflow.saves`. PF2e has no equivalent reliable source in the `createChatMessage` payload.

**Current impact:** For AoE spells, PF2e will typically resolve 0-1 targets (falling to the "no targets resolved" branch which plays a generic `BLOODY_HIT`). Single-target attacks work correctly when the user has a target selected.

**Possible improvement:** PF2e v5+ exposes `game.pf2e.effectPanel` and `ChatMessage#targets` in some versions. Investigation needed for a better multi-target API.

#### A1-5: VOCAL_STAGGER Default (1400ms)
**Severity: 🔵 INFO**

PF2e's `VOCAL_STAGGER` defaults to 1400ms (via `SoundOrchestrator.DEFAULT_OFFSETS.VOCAL_STAGGER`). DnD5e's `handleDamage` also reads from the orchestrator with fallback 400ms.

**Analysis:** Looking at the code:
- DnD5e: `orch?.getNamedOffset("VOCAL_STAGGER") ?? 400` — fallback 400ms if orchestrator unavailable
- PF2e: `orch?.getNamedOffset("VOCAL_STAGGER") ?? 1400` — fallback 1400ms if orchestrator unavailable

When the orchestrator IS available, both systems use the same configured value (default 1400ms from `DEFAULT_OFFSETS`). The difference in hardcoded fallbacks only matters if the orchestrator fails to load.

**Status:** The 1400ms orchestrator default is intentional and shared. The DnD5e hardcoded fallback of 400ms is tighter because Midi-QOL damage hooks fire with precise timing. PF2e's 1400ms fallback accounts for the delay between `createChatMessage` and target HP actually updating. This is correct design.

### Detailed Daggerheart Findings

#### A2-1: Two-Phase Render Timing
**Severity: 🟡 COSMETIC**

Phase 2 triggers when `(Date.now() - phase.timestamp) > 500`. If DH system changes its re-render timing (currently ~4s after initial render), the threshold could:
- **False positive**: If DH renders quickly (<500ms), Phase 2 won't trigger (result sounds silent).
- **Double-fire**: If DH adds a third render pass, Phase 2 could fire twice (phase tracking prevents this — once set to phase 2, further renders are skipped).

**Current risk:** Low. The 500ms threshold is generous. DH's current re-render happens at ~4 seconds. A more robust approach would be to detect specific DOM content changes rather than relying on timing alone.

#### A2-2: Content Keyword Matching
**Severity: 🟠 WRONG**

Result detection relies on `msgContains(content, ["MISS", "FAILURE", "FAIL"])` and similar for success. This scans the rendered HTML text content for uppercase keywords.

**Problems:**
- **False positives**: A message containing "DISMISS" would match "MISS". A message about "WATERFALL" would match "FAIL". The `msgContains` utility should be verified for word-boundary matching.
- **Localization**: If DH system localizes to non-English, these keywords won't match.
- **Format changes**: If DH changes result display format (e.g. from "MISS" to "Miss" or uses icons instead of text), detection breaks silently.

**Mitigating factor**: The primary detection path uses DC comparison and flags. Content scanning is a last-resort fallback when flags/DC are absent.

#### A2-3: renderChatMessageHTML Hook Name
**Severity: 🔵 INFO**

Comment states: "renderChatMessageHTML replaces renderChatMessage (deprecated v13, removed v15)."

**Analysis:** The adapter uses `renderChatMessageHTML` which is the correct hook for Foundry VTT v12+ (current target). For Foundry v11 compatibility, `renderChatMessage` would be needed. The comment accurately reflects the deprecation timeline. Current code is correct for Foundry v12/v13.

#### A2-4: renderPhases Memory Leak
**Severity: 🟡 COSMETIC**

`this.renderPhases` is a `Map()` initialized in the constructor. Entries are added on every chat message render but never cleaned up.

**Impact:** Each entry stores `{ timestamp, phase, data }` where `data` includes roll info, item references, etc. Over a 6+ hour session with frequent rolling (say 200+ rolls), this accumulates ~200 Map entries with moderate object graphs. Memory impact is low (a few KB) but unbounded growth is poor practice.

**Fix:** Add cleanup when Phase 2 completes (delete entry), or implement a periodic sweep (delete entries older than 30 seconds).

#### A2-5: Item Resolution Chain Fragility
**Severity: 🟠 WRONG**

The 4-step resolution chain in `handleInfo()`:
1. UUID via `roll.options.itemUuid || roll.options.item?.uuid || message.flags.daggerheart.itemUuid || message.flags.core.itemUuid`
2. Exact name match: `actor.items.getName(itemName)`
3. Fuzzy prefix: `actor.items.find(i => i.name.toLowerCase().startsWith(lower))`
4. Raw title match: `actor.items.getName(rawTitle)`

**Problems:**
- Step 3 (fuzzy prefix) can match wrong items. E.g. "Fire" prefix matches "Fireball" AND "Fire Shield".
- Title parsing uses `:` and ` - ` as separators, but DH system format may evolve.
- If UUID resolution becomes available (DH adds it to roll payloads), steps 2-4 become dead code that could still cause mismatches if UUID lookup fails.

**Current state:** This works for current DH beta versions where UUIDs are not consistently present. The code correctly prioritizes UUID when available.

### Critical Gaps from Leaks

1. **🔴 PF2e has no spell effect-type config section.** `SoundConfigApp.js:326` only renders spell children (SPELL_FIRE, SPELL_ICE, etc.) for DnD5e. PF2e users see "Magic (Spells)" with zero children — they cannot configure the same SPELL_* keys that the PF2e adapter actually resolves via `_getSpellTraitKey()`. The runtime works (trait → key → fallback), but users have no UI to customize these bindings.

2. **🔴 PF2e has no roll stinger config.** `SoundConfigApp.js:619-676` only adds roll stinger sections for DH and DnD5e. PF2e's `_handleOutcome()` plays `ROLL_CRIT` and `ROLL_FUMBLE` at runtime, but these keys are invisible in the config UI for PF2e users. They can only be configured via JSON import/export.

3. **🟡 ActorSoundConfig doesn't know about PF2e-specific slots.** Falls to generic DnD5e `else` branch (only "Your Turn" spotlight). PF2e could benefit from Hero Point gain/spend sound overrides.
